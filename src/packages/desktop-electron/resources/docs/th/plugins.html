ปลั๊กอิน

เขียนปลั๊กอินของคุณเองเพื่อขยาย vanilla-closedcode

ปลั๊กอินช่วยให้คุณสามารถขยาย vanilla-closedcode ได้โดยเชื่อมโยงกับเหตุการณ์ต่างๆ และปรับแต่งลักษณะการทำงาน คุณสามารถสร้างปลั๊กอินเพื่อเพิ่มคุณสมบัติใหม่ บูรณาการกับบริการภายนอก หรือแก้ไขการทำงานเริ่มต้นของ vanilla-closedcode

ตัวอย่างเช่น ลองดู ปลั๊กอิน ที่สร้างโดยชุมชน

ใช้ปลั๊กอิน

มีสองวิธีในการโหลดปลั๊กอิน

จากไฟล์ในเครื่อง

วางไฟล์ JavaScript หรือ TypeScript ในไดเร็กทอรีปลั๊กอิน

  • .closedcode/plugins/ - ​​ปลั๊กอินระดับโครงการ
  • ~/.config/closedcode/plugins/ - ​​ปลั๊กอินทั่วโลก

ไฟล์ในไดเร็กทอรีเหล่านี้จะถูกโหลดโดยอัตโนมัติเมื่อเริ่มต้นระบบ

จาก npm

ระบุแพ็คเกจ npm ในไฟล์ปรับแต่งของคุณ

{
  "plugin": ["opencode-helicone-session", "opencode-wakatime", "@my-org/custom-plugin"]
}

รองรับแพ็คเกจ npm ทั้งแบบปกติและแบบกำหนดขอบเขต

เรียกดูปลั๊กอินที่มีอยู่ใน ระบบนิเวศ

วิธีการติดตั้งปลั๊กอิน

ปลั๊กอิน npm ได้รับการติดตั้งโดยอัตโนมัติโดยใช้ npm เมื่อเริ่มต้น แพ็คเกจและการขึ้นต่อกันถูกแคชไว้ใน ~/.cache/closedcode/node_modules/

ปลั๊กอินท้องถิ่น โหลดโดยตรงจากไดเร็กทอรีปลั๊กอิน หากต้องการใช้แพ็คเกจภายนอก คุณต้องสร้าง package.json ภายในไดเร็กทอรีการกำหนดค่าของคุณ (ดู การพึ่งพา) หรือเผยแพร่ปลั๊กอินไปที่ npm และ เพิ่มลงในการกำหนดค่าของคุณ

โหลดออเดอร์

ปลั๊กอินถูกโหลดจากทุกแหล่งและ hooks ทั้งหมดทำงานตามลำดับ ลำดับการโหลดคือ:

  1. การกำหนดค่าส่วนกลาง (~/.config/closedcode/closedcode.json)
  2. การกำหนดค่าโครงการ (closedcode.json)
  3. ไดเร็กทอรีปลั๊กอินสากล (~/.config/closedcode/plugins/)
  4. ไดเรกทอรีปลั๊กอินโครงการ (.closedcode/plugins/)

แพ็กเกจ npm ที่ซ้ำกันซึ่งมีชื่อและเวอร์ชันเดียวกันจะถูกโหลดหนึ่งครั้ง อย่างไรก็ตาม ทั้งปลั๊กอินในเครื่องและปลั๊กอิน npm ที่มีชื่อคล้ายกันจะโหลดแยกกัน

สร้างปลั๊กอิน

ปลั๊กอินคือ JavaScript/TypeScript module ที่ส่งออกปลั๊กอินอย่างน้อยหนึ่งรายการ ฟังก์ชั่น แต่ละฟังก์ชันได้รับวัตถุบริบทและส่งกลับวัตถุ hooks

การพึ่งพาอาศัยกัน

ปลั๊กอินท้องถิ่นและเครื่องมือแบบกำหนดเองสามารถใช้แพ็คเกจ npm ภายนอกได้ เพิ่ม package.json ลงในไดเร็กทอรี config ของคุณด้วยการอ้างอิงที่คุณต้องการ

{
  "dependencies": {
    "shescape": "^2.1.0"
  }
}

vanilla-closedcode รัน npm install เมื่อเริ่มต้นเพื่อติดตั้งสิ่งเหล่านี้ ปลั๊กอินและเครื่องมือของคุณสามารถนำเข้าได้




  return {
    "tool.execute.before": async (input, output) => {
      if (input.tool === "bash") {
        output.args.command = escape(output.args.command)
      }
    },
  }
}

โครงสร้างพื้นฐาน


  console.log("Plugin initialized!")

  return {
    // Hook implementations go here
  }
}

ฟังก์ชั่นปลั๊กอินได้รับ:

  • project: ข้อมูลโครงการปัจจุบัน
  • directory: ไดเร็กทอรีการทำงานปัจจุบัน
  • worktree: เส้นทางเวิร์กทรีคอมไพล์
  • client: ไคลเอนต์ closedcode SDK สำหรับการโต้ตอบกับ AI

รองรับ TypeScript

สำหรับปลั๊กอิน TypeScript คุณสามารถนำเข้าประเภทจากแพ็คเกจปลั๊กอินได้:




  return {
    // Type-safe hook implementations
  }
}

ตัวอย่าง

นี่คือตัวอย่างบางส่วนของปลั๊กอินที่คุณสามารถใช้เพื่อขยาย closedcode

ส่งการแจ้งเตือน

ส่งการแจ้งเตือนเมื่อมีเหตุการณ์บางอย่างเกิดขึ้น:


  return {
    event: async ({ event }) => {
      // Send notification on session completion
      if (event.type === "session.idle") {
        await $`osascript -e 'display notification "Session completed!" with title "closedcode"'`
      }
    },
  }
}

เรากำลังใช้ osascript เพื่อเรียกใช้ AppleScript บน macOS ที่นี่เราใช้มันเพื่อส่งการแจ้งเตือน

หากคุณใช้แอปเดสก์ท็อป vanilla-closedcode แอปสามารถส่งการแจ้งเตือนของระบบโดยอัตโนมัติเมื่อมีการตอบกลับพร้อมหรือเมื่อเซสชันเกิดข้อผิดพลาด
---

การป้องกัน .env

ป้องกันไม่ให้ closedcode อ่านไฟล์ .env:


  return {
    "tool.execute.before": async (input, output) => {
      if (input.tool === "read" && output.args.filePath.includes(".env")) {
        throw new Error("Do not read .env files")
      }
    },
  }
}

ฉีดตัวแปรสภาพแวดล้อม

แทรกตัวแปรสภาพแวดล้อมลงในการดำเนินการ shell ทั้งหมด (เครื่องมือ AI และ terminal ผู้ใช้):


  return {
    "shell.env": async (input, output) => {
      output.env.MY_API_KEY = "secret"
      output.env.PROJECT_ROOT = input.cwd
    },
  }
}

เครื่องมือที่กำหนดเอง

ปลั๊กอินยังสามารถเพิ่มเครื่องมือที่กำหนดเองให้กับ closedcode:




  return {
    tool: {
      mytool: tool({
        description: "This is a custom tool",
        args: {
          foo: tool.schema.string(),
        },
        async execute(args, context) {
          const { directory, worktree } = context
          return `Hello ${args.foo} from ${directory} (worktree: ${worktree})`
        },
      }),
    },
  }
}

ตัวช่วย tool สร้างเครื่องมือแบบกำหนดเองที่สามารถเรียกใช้ closedcode ได้ ใช้ฟังก์ชัน Zod schema และส่งกลับคำจำกัดความของเครื่องมือด้วย:

  • description: เครื่องมือนี้ทำอะไร
  • args: Zod schema สำหรับข้อโต้แย้งของเครื่องมือ
  • execute: ฟังก์ชั่นที่ทำงานเมื่อมีการเรียกใช้เครื่องมือ

เครื่องมือที่คุณกำหนดเองจะพร้อมใช้งานสำหรับ closedcode ควบคู่ไปกับเครื่องมือในตัว

หากเครื่องมือปลั๊กอินใช้ชื่อเดียวกับเครื่องมือในตัว เครื่องมือปลั๊กอินจะมีความสำคัญเหนือกว่า
---

การบันทึก

ใช้ client.app.log() แทน console.log สำหรับการบันทึกแบบมีโครงสร้าง:


  await client.app.log({
    body: {
      service: "my-plugin",
      level: "info",
      message: "Plugin initialized",
      extra: { foo: "bar" },
    },
  })
}

ระดับ: debug, info, warn, error ดูSDKเอกสารประกอบสำหรับรายละเอียด

Hooks สำหรับการกระชับข้อมูล

ปรับแต่งบริบทที่รวมไว้เมื่อมีการกระชับเซสชัน:




  return {
    "experimental.session.compacting": async (input, output) => {
      // Inject additional context into the compaction prompt
      output.context.push(`
## Custom Context

Include any state that should persist across compaction:
- Current task status
- Important decisions made
- Files being actively worked on
`)
    },
  }
}

experimental.session.compacting hook เริ่มทำงานก่อนที่ LLM จะสร้างข้อมูลสรุปต่อเนื่อง ใช้เพื่อแทรกบริบทเฉพาะโดเมนที่พรอมต์การบีบอัดข้อมูลเริ่มต้นอาจพลาดไป

คุณยังสามารถแทนที่พรอมต์การกระชับข้อมูลทั้งหมดได้โดยตั้งค่า output.prompt:




  return {
    "experimental.session.compacting": async (input, output) => {
      // Replace the entire compaction prompt
      output.prompt = `
You are generating a continuation prompt for a multi-agent swarm session.

Summarize:
1. The current task and its status
2. Which files are being modified and by whom
3. Any blockers or dependencies between agents
4. The next steps to complete the work

Format as a structured prompt that a new agent can use to resume work.
`
    },
  }
}

เมื่อตั้งค่า output.prompt มันจะแทนที่พร้อมท์การกระชับค่าเริ่มต้นโดยสมบูรณ์ อาร์เรย์ output.context จะถูกละเว้นในกรณีนี้
