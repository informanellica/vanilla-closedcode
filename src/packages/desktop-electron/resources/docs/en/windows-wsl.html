Windows (WSL)

Run vanilla-closedcode on Windows using WSL for the best experience.

While vanilla-closedcode can run directly on Windows, we recommend using Windows Subsystem for Linux (WSL) for the best experience. WSL provides a Linux environment that works seamlessly with vanilla-closedcode's features.

Why WSL?

WSL offers better file system performance, full terminal support, and compatibility with development tools that vanilla-closedcode relies on.
---

Setup

  1. Install WSL

    If you haven't already, install WSL using the official Microsoft guide.

  2. Install vanilla-closedcode in WSL

    Once WSL is set up, open your WSL terminal and download closedcode from the Releases page, then install the downloaded build inside WSL.

  3. Use vanilla-closedcode from WSL

    Navigate to your project directory (access Windows files via /mnt/c/, /mnt/d/, etc.) and run vanilla-closedcode.

    cd /mnt/c/Users/YourName/project
closedcode

Desktop App + WSL Server

If you prefer using the vanilla-closedcode Desktop app but want to run the server in WSL:

  1. Start the server in WSL with --hostname 0.0.0.0 to allow external connections:

    closedcode serve --hostname 0.0.0.0 --port 4096

  2. Connect the Desktop app to http://localhost:4096

If `localhost` does not work in your setup, connect using the WSL IP address instead (from WSL: `hostname -I`) and use `http://:4096`.
When using `--hostname 0.0.0.0`, set `CLOSEDCODE_SERVER_PASSWORD` to secure the server.
```bash CLOSEDCODE_SERVER_PASSWORD=your-password closedcode serve --hostname 0.0.0.0 ```

Web Client + WSL

For the best web experience on Windows:

  1. Run closedcode web in the WSL terminal rather than PowerShell:

    closedcode web --hostname 0.0.0.0

  2. Access from your Windows browser at http://localhost:<port> (vanilla-closedcode prints the URL)

Running closedcode web from WSL ensures proper file system access and terminal integration while still being accessible from your Windows browser.

Accessing Windows Files

WSL can access all your Windows files through the /mnt/ directory:

  • C: drive → /mnt/c/
  • D: drive → /mnt/d/
  • And so on...

Example:

cd /mnt/c/Users/YourName/Documents/project
closedcode
For the smoothest experience, consider cloning/copying your repo into the WSL filesystem (for example under `~/code/`) and running vanilla-closedcode there.
---

Tips

  • Keep vanilla-closedcode running in WSL for projects stored on Windows drives - file access is seamless
  • Use VS Code's WSL extension alongside vanilla-closedcode for an integrated development workflow
  • Your vanilla-closedcode config and sessions are stored within the WSL environment at ~/.local/share/closedcode/
